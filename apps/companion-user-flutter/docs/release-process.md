## 1. Version numbering

Format: `MAJOR.MINOR.PATCH+BUILD` (Flutter `pubspec.yaml` convention).

| Field | Rule |
|-------|------|
| MAJOR | Breaking change to the companion API or complete rework |
| MINOR | New user-visible feature; backwards compatible |
| PATCH | Bug fix or non-functional change (docs, deps, perf) |
| BUILD | Auto-incremented by CI (`${{ github.run_number }}`) |

Example: `1.4.2+87`

Tags follow the pattern `flutter-user-vMAJOR.MINOR.PATCH` (e.g. `flutter-user-v1.4.2`).

## 2. Release checklist

Run through these steps before tagging a release.

- [ ] `main` is green (all CI jobs pass on latest commit).
- [ ] `flutter analyze` reports zero issues locally.
- [ ] `flutter test` — all tests pass, no skips without justification.
- [ ] `dart format --output=none --set-exit-if-changed .` — no formatting drift.
- [ ] Version bumped in `pubspec.yaml` (`version: X.Y.Z+BUILD`).
- [ ] `CHANGELOG.md` reviewed (auto-generated by `.github/workflows/changelog.yml` on tag push).
- [ ] Sentry release created (automatic via CI if `SENTRY_AUTH_TOKEN` secret is set).
- [ ] No TODO / FIXME comments in code paths targeted by this release.
- [ ] QA sign-off on physical device (R58N94KML7J or equivalent).

```
                                       build signed AAB (arm64, prod)

```

## 4. Cutting a release tag

```bash
# 1. Ensure local main is up-to-date.
git checkout main && git pull

# 2. Bump version in pubspec.yaml (edit manually or use sed).
#    Do NOT change the +BUILD suffix — CI handles that automatically.
sed -i 's/^version: .*/version: X.Y.Z/' \
  apps/companion-user-flutter/pubspec.yaml

# 3. Commit the version bump.
git add apps/companion-user-flutter/pubspec.yaml
git commit -m "chore(release): bump version to X.Y.Z"

# 4. Push and verify CI is green.
git push origin main

# 5. Tag and push.
git tag flutter-user-vX.Y.Z
git push origin flutter-user-vX.Y.Z
```

This triggers:

- `changelog.yml` → auto-updates `CHANGELOG.md` on `main`.
- `firebase-app-distribution.yml` → builds signed APK + uploads to internal testers.

## 5. Firebase App Distribution (internal testers)

Handled automatically by `.github/workflows/firebase-app-distribution.yml` on every tag push.

**Manual upload** (when CI has no keystore secrets):

```bash
# Build locally (arm64 release):
cd apps/companion-user-flutter
flutter build apk --release --target-platform android-arm64

# Upload with Firebase CLI:
firebase appdistribution:distribute \
  build/app/outputs/flutter-apk/app-release.apk \
  --app "$FIREBASE_APP_ID" \
  --groups "internal-testers" \
  --release-notes "$(git log -5 --pretty=format:'• %s')"
```

**Required secrets** (GitHub → Settings → Secrets and variables → Actions):

| Secret | Description |
|--------|-------------|
| `FIREBASE_APP_ID` | Firebase Android App ID (`1:NNNN:android:XXXX`) |
| `FIREBASE_TOKEN`  | Token from `firebase login:ci` |
| `KEYSTORE_BASE64` | `base64 -w0 release.keystore` |
| `KEYSTORE_PASSWORD` | Store password |
| `KEY_ALIAS` | Key alias in keystore |
| `KEY_PASSWORD` | Key password |

## 6. Play Store submission

Automated via `.github/workflows/flutter-user-release.yml`, triggered by every
`flutter-user-v*` tag push. The workflow:

1. Decodes the release keystore from `KEYSTORE_BASE64` secret.
2. Writes `android/key.properties` so Gradle picks up the release signing config.
3. Runs `flutter test --exclude-tags golden` (must pass before build proceeds).
4. Builds a signed AAB: `flutter build appbundle --flavor prod --target lib/main.dart`.
5. Uploads to the **internal** track via [`r0adkll/upload-google-play`](https://github.com/r0adkll/upload-google-play).

Use `workflow_dispatch` to target `alpha`, `beta`, or `production` tracks manually.

Before running a local or manual release, generate the current operator checklist:

```bash
cd ../../
npm run ops:release:prep-all
```

For a mobile-only view, use:

```bash
npm run ops:mobile:prep-release-evidence
```

`ops:release:prep-all` prints the combined release picture:

- release metadata (`release_id`, `head_sha`, `artifact_manifest_hash`)
- current strict blockers across checklist / soak / mobile / final signoff
- execution model for which sections can run independently
- mobile evidence refresh commands
- mobile define-file target / backup / rollback details
- per-role final signoff failing checks and emitted signoff commands
- a generated handoff script: `docs/release/status/release-ops-next-steps.ps1`

When soak is already running, the preferred execution order from that combined handoff is:

1. rollout
2. mobile evidence
3. final signoff
4. final strict status rerun

The generated `release-ops-next-steps.ps1` file is intended to be edited in place:

- fill the variables at the top
- keep the generated commands below unchanged unless the release workflow itself changes
- only the mobile sections require the mobile variables to be filled; soak commands can run without the mobile/signoff placeholders
- if you use the generated mobile define-file update step, it creates `config/env/mobile-dart-defines.release.local.json.bak` first
- the script prints the exact rollback command for restoring that backup
- the current RC mobile scope is `android-only`; iOS is explicitly `deferred`

`ops:mobile:prep-release-evidence` prints:

- current mobile readiness blockers
- artifact ages
- the exact refresh commands for validation/signing/crash/privacy/perf gates
- source materials for the current mobile evidence lane
- input mapping showing which operator variable or action resolves each failing mobile check
- blocker-aware next steps ordered to match the current mobile blocker set
- Firebase/TLS define hints from the active release define bundle
- the active `define_bundle_source`
- repo-observed Android Firebase values that can be reused directly
- the exact missing release-only values that still require operator input
- persisted mobile prep artifacts:
  - `docs/release/status/mobile-release-prep-latest.json`
  - `docs/release/status/mobile-release-prep-latest.md`
- `docs/release/status/mobile-release-next-steps.ps1`
- the dedicated mobile handoff script is intended to be edited in place:
  - fill the mobile variables at the top
  - only the mobile sections validate those mobile inputs
  - rollout and signoff placeholders are not required there
  - if it updates `config/env/mobile-dart-defines.release.local.json`, it creates a `.bak` backup first and prints the rollback command
  - for the current Android-only RC it does not require `IosBuildRef` or iOS signing inputs

**Required secrets:**

| Secret | Description |
|--------|-------------|
| `KEYSTORE_BASE64` | `base64 -w0 release.keystore` encoded keystore |
| `KEYSTORE_PASSWORD` | Store password |
| `KEY_ALIAS` | Key alias inside the keystore |
| `KEY_PASSWORD` | Key password |
| `PLAY_STORE_SERVICE_ACCOUNT_JSON` | Google Cloud service account JSON with *Release Manager* role in Play Console |

**Local build** (for testing signing without pushing a tag):

```bash
# 1. Copy and fill in the keystore template:
cp android/key.properties.example android/key.properties
# edit android/key.properties with real values

# 2. Generate release dart-defines file from environment variables:
#    (set required SVEN_* env vars first; see below)
cd ../../
npm run mobile:dart-defines:generate:strict -- --out config/env/mobile-dart-defines.release.local.json

# 3. Build signed AAB:
cd apps/companion-user-flutter
flutter build appbundle \
  --release \
  --flavor prod \
  --target lib/main.dart \
  --target-platform android-arm64 \
  --dart-define-from-file=../../config/env/mobile-dart-defines.release.local.json

# Signed AAB location:
# build/app/outputs/bundle/prodRelease/app-prod-release.aab
```

Required environment variables for step 2:

- `SVEN_API_CERT_SHA256_PINS` (comma/space-separated SHA-256 cert pins)
- `SVEN_FIREBASE_IOS_PROD_API_KEY`
- `SVEN_FIREBASE_IOS_PROD_APP_ID`
- `SVEN_FIREBASE_IOS_PROD_BUNDLE_ID`
- `SVEN_FIREBASE_WEB_API_KEY`
- `SVEN_FIREBASE_WEB_APP_ID`

Current local release-define behavior:

- `node scripts/mobile-firebase-tls-define-check.cjs --strict` now accepts values from:
  1. exported shell env
  2. `config/env/mobile-dart-defines.ci.release.json`
  3. `config/env/mobile-dart-defines.release.local.json`
- The checker report includes `define_bundle_source` so operators can see which file was actually used.

Current locally missing values are expected to be filled from real release metadata:

- `SVEN_API_CERT_SHA256_PINS`
- `SVEN_FIREBASE_IOS_PROD_APP_ID`
- `SVEN_FIREBASE_WEB_APP_ID`

Values that are already present in the local release define file or repo can be reused as-is:

- `SVEN_FIREBASE_IOS_PROD_API_KEY`
- `SVEN_FIREBASE_IOS_PROD_BUNDLE_ID`
- `SVEN_FIREBASE_WEB_API_KEY`
- Android Firebase app id/package values from `google-services.json`

**Pre-launch checklist for Play Store:**

- Privacy policy URL up-to-date.
- Content rating questionnaire completed.
- App permissions description matches actual usage.
- Screenshots provided for all required form factors.
- `applicationId` changed from `com.example.sven_user_flutter` to the real package name before first submission.

## 7. App Store submission (iOS)

1. Build on macOS with signing configured:

   ```bash
   flutter build ios --release \
     --build-number "$CI_BUILD_NUMBER"
   ```

2. Archive in Xcode (`Product → Archive`).
3. Upload via Xcode Organiser or `xcrun altool`.
4. Submit for TestFlight then App Store Review.

Required: Apple Developer account with Distribution certificate + App Store provisioning profile (see `.github/workflows/flutter-user-app-ci.yml` for CI-side signing setup).

## 8. Hotfix procedure

```
main (broken)
  └── hotfix/X.Y.(PATCH+1)
        • fix the issue
        • bump version patch
        • flutter test (locally)
        • PR → main
        • merge + tag flutter-user-vX.Y.(PATCH+1)
```

**Rules:**

- Hotfix branches should contain only the minimal change.
- Must include a regression test.
- Do NOT merge unrelated improvements.
- Tag immediately after merge — do not let hotfix sit untagged.
